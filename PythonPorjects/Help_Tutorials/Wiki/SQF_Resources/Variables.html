    Variables are used to store data (e.g. numbers, text, or even objects). Different entities in VBS (e.g. scripts, triggers, objects & addons) can read and modify these variables, and thus use them to communicate with each other.

    Certain rules exist in regards to naming of variable. See the article on Identifier for details.

    Namespaces

    Variables are only visible in certain areas (namespaces) of the game. This prevents name conflicts between different variables in different Script (File).

    Local

    Local variables are only visible in a specific Script (File). The whole code in the script has access to the variable, including any Function it may call.
    User-defined local variables cannot be used in init lines, triggers or waypoints, etc., but they can be used in EXEC.
    Some engine-defined local variables (e.g. this or onMapSingleClick) may be available in specific situations.

    The Identifier of local variables always has to start with an underscore.

    _myLocalVar = 0;

    Global

    Global variables are visible on the whole computer where they are defined.

    Names given to entities (units, vehicles, objects, etc) in the Mission Editor are also global variables, and should not be modified.

    Identifiers of global variables must not start with underscore. Otherwise the same Identifier apply.

    myGlobalVariable = 0;

    Public

    Public variables are global variables which are available on all computers in the network.

    The value of a global variable gets broadcasted over the network using publicVariable. After the call of this command the variable will have the same value on all clients. Every time the value of this variable changes, it will have to be re-broadcast via another publicVariable execution.

    publicVariable "myPublicVariable";

    Initialization

    Variables don't need to be explicitly declared before their first use. The VBS engine will automatically initialize them, as soon as a value is assigned to them.

    Querying undefined (or uninitialized) variables returns nil (undefined value). When converted to String with str, they will always return scalar bool array string 0xe0ffffef. In-game the command isNil can be used to check whether a variable has been declared yet.

    Scope

    Declarations

    Once a global variable has received a value it is accessible anywhere, no matter where the initialization happened (e.g. whether it'd be in a script, a function, an object's init line, or a trigger).

    Local variables can be initialized via an assignment or via the private command (this is the recommended practice for Functions). Once initialized, local variables are only available on the same or a lower level scope:

     _txt = "hello";
     if (alive player) then {hint _txt};

    Since V3.7 this automatic inheritance behavior can be overridden via the privateAll command.

    If a local variable is assigned in a lower scope though (e.g. within a Control Structures), then, by default, it is not available to higher scopes:

     waitUntil {
       _time = systemTime; // '_time' is assigned within control structure (lower level scope)
       !alive tank1
     };
     // variable '_time' is undefined at this point
     hint format["Tank was killed @ %1:%2 ",_time select 3,_time select 4];

    In order to read the value that was assigned at a lower level, the variable must be initialized beforehand (on the scope level it is supposed to be read later on), either via the private command, or by assigning a value to it:

     private ["_time"]; // initialization via 'private'
     waitUntil {
       _time = systemTime;
       !alive tank1
     };
     hint format["Tank was killed @ %1:%2 ",_time select 3,_time select 4];
     _time = []; // alternative initialization via assignment
     waitUntil {
       _time = systemTime;
       !alive tank1
     };
     hint format["Tank was killed @ %1:%2 ",_time select 3,_time select 4];

    Functions

    Functions that are called from a script are considered to be on a lower scope, and share the variable space:

    _fnc = {
      _i = 9; // _fnc sets '_i' to 9
      player sidechat format["in function: %1",_i];
    };

    _i = 1;
    [] call _fnc; 
    player sidechat format["after function: %1",_i]; // '_i' now contains 9

    To prevent functions from overwriting variables, the private command should be used to declare all variables used within that function:

    _fnc = {
      private ["_i"]; // a new and separate instance of '_i' is created, not affecting the existing one
      _i = 9; // _fnc sets '_i' to 9, but only within its scope
      player sidechat format["in function: %1",_i];
    };

    _i = 1;
    [] call _fnc; 
    player sidechat format["after function: %1",_i]; // '_i' still contains 1

    Loops

    The scope of index variables used in for depends on the syntax of the loop.
    The for var variation will create a new instance of the index variable, which will not affect one of the same name, that might exist at a higher scope:

    _i = -1;
    for "_i" from 0 to 3 do {
      hint str _i;
    };
    hint format["[K] after 'for...do': %1",_i]; // '_i' is still -1, the for...do loop didn't affect it

    The for forspec variation though, will modify the variable if it exists already

    _i = -1;
    for [{_i=0},{_i<3},{_i=_i+1}] do {
      hint str _i;
    };
    hint format["[K] after 'for [...] do': %1",_i]; // '_i' is now 3 (last value in the loop)

    In either case, if the variable didn't exist before the loop, it will still be undefined after the loop:

    for "_i" from 0 to 3 do {
      hint str _i;
    };
    if (isNil "_i") then {player sidechat "undefined"}; // '_i' is still undefined

    for [{_i=0},{_i<3},{_i=_i+1}] do {
      hint str _i;
    };
    if (isNil "_i") then {player sidechat "undefined"}; // '_i' is still undefined

    Data Types

    Deletion

    Once created, variables will take up space in your computer memory. Local variables are destroyed as soon as their initiating script exist, but if large global variables are used, they should be manually undefined once they are not needed anymore. This can be done by assigning nil to it:

    hugeVariable = nil;
